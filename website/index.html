Why Valor exists

Most career tooling sees only the visible artifacts.

Commits miss the real work

Debugging, design thinking, mentoring, cross-team alignment, and operational ownership rarely show up cleanly in merge history.

Manual brag docs decay fast

By review season, developers are reconstructing months of work from memory instead of building evidence as the work happens.

Hosted career tools create trust drag

When a tool sits near your code, tickets, and assistant conversations, privacy and inspectability matter as much as features.

How it works

A local core that fits the way developers already work.

01

Work through your assistant

Valor rides alongside assistant-led developer workflows instead of asking for a second system of record.

02

Store evidence locally

State, framework, evidence, and carry-forward notes live in local files under ~/.valor/.

03

Map work to your ladder

Valor uses your own career framework instead of imposing a generic ladder or HR taxonomy.

04

Nudge the next-level move

After meaningful work, Valor can connect what happened to target-level behaviors and suggest one concrete way to go further.

Privacy and trust

Local-first is a product choice, not just a tagline.

What this repo does

  • Stores local core state under ~/.valor/
  • Keeps prompts, rules, and logic inspectable
  • Runs without built-in telemetry or cloud sync
  • Lets users bring their own career framework

What this repo does not do

  • No hosted backend
  • No mandatory account
  • No built-in analytics pipeline
  • No hidden employer-facing dashboard

Trust boundary

Valor’s local core is inspectable, but the surrounding assistant, model provider, and integrations still matter. If your host assistant sends prompts or workspace context to a hosted model, that behavior is governed outside this repo.
